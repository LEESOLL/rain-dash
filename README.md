## 기획 의도 — "속도"에 대한 해석

"속도"를 주제로 웹게임을 만든다고 했을 때, 가장 먼저 떠오른 건 빠르게 달리는 런게임이나 레이싱 게임이었다. 그런데 그건 누구나 가장 먼저 떠올릴 법한 답이기도 했다. 그래서 속도를 조금 다른 각도에서 보고 싶었다.

속도는 보통 '공간을 얼마나 빨리 가로지르는가'로 잰다. 하지만 빠르다·느리다를 실제로 가르는 건 시간이다 — 같은 시간에 얼마나 멀리 갔느냐. 그래서 빠르기 대신 **시간 그 자체**를 플레이어 손에 쥐어주기로 했다. 내가 움직일 때만 세상의 시간이 흐르고, 멈추면 세상도 멈춘다면?

이 아이디어를 어디에 얹을지 고민하다, 비가 떠올랐다.

> 우산 없이 집 가는 비 오는 날.
> _"빨리 뛰면 더 젖을까, 천천히 걸으면 더 젖을까?"_

누구나 한 번쯤 해본 고민이다. 정답은 나도 모른다. 하지만 이 물음 안에는 **속도가 곧 '얼마나 젖느냐'를 바꾼다는**, 모두가 몸으로 아는 속도의 문제가 들어 있다.

그렇게 **Rain Dash**의 윤곽을 잡았다. 이 게임은 속도에 한 가지로 답하지 않는다. 세 가지를 한꺼번에 답한다.

- **빠름** — 빨리 달릴수록 집은 가까워진다. 대신 비도 번개도 거세진다. 짜릿한 만큼 대가가 따른다.
- **느림** — 멈추면 시간이 느려지고, 번개를 읽고 피할 수 있다. 멈출 줄 아는 것도 속도다.
- **전혀 다른 것** — 빠른 건 내가 아니라 세상이다. 그 템포를 내가 정한다.

## How To Play

- **목표**: 비, 웅덩이, 번개를 피하며 집(골인 지점)까지 도착하면 클리어.
- **핵심 메커닉 (시간 스케일)**: 내가 움직일 때만 시간이 정상 속도로 흐른다. 멈추면 비·번개가 느려진다. → **속도를 내가 직접 조절**.
- **조작**
  - 데스크탑: `←` `→` `a` `d` 이동 / `↑`·`w`·`Space` 점프(2단 점프)
  - 모바일: 화면 3등분 — 왼쪽=뒤로, 오른쪽=앞으로, 가운데=점프 (멀티터치로 달리며 점프 가능)
- **위협**: 비 · 번개(경고 후 낙뢰, 빠를수록 못 피함) · 물웅덩이(점프로 회피) → 목숨 -1
- **아이템**: 하트(목숨 회복) · 우산(비 방어) · 장화(웅덩이 방어) · 우비(둘 다 방어)
- **쉘터 · 집 지붕**: 가게·버스정류장·공중전화, 그리고 골인 지점의 집 — 지붕 아래로 들어가면 비를 막아줌(단, 번개는 못 막음)

## 점수 — "속도"를 보상으로 번역하기

점수 시스템은 게임의 주제인 **속도**를 그대로 담는 그릇이어야 했다. 이 게임에선 내가 움직일 때만 시간이 정상 속도로 흐른다 — 달리면 비도 번개도 제 속도로 몰아치고, 멈추면 세상이 함께 느려져 번개를 읽고 피할 여유가 생긴다. 즉 **달리는 만큼 위험해지고, 멈추는 만큼 안전해진다.**

여기에 점수를 얹었다. 빠르게 달릴수록 점수 배수가 최대 5배까지 오르고, 멈춰서 안전하게 가면 1배로 내려간다. 그러면 위험을 키우는 손잡이와 점수를 키우는 손잡이가 따로 있지 않고 **같은 하나(=움직임)** 가 된다. 그래서 플레이어는 매 순간 "지금 달려서 점수를 벌까(위험), 멈춰서 안전하게 갈까(저득점)"를 직접 선택하게 된다.

- **거리 × 속도 배수** — 같은 거리를 가도 빠르게 달릴 때 점수가 최대 5배까지 쌓인다. 멈춰서 안전하게 가면 점수도 천천히 오른다. → **위험을 감수한 속도에 보상**.
- **클리어 보너스** — 빨리 도착할수록(시간 보너스), 안 다칠수록(잔여 목숨 보너스) 가산. 속도와 생존을 동시에 보상.
- **베스트 점수 · NEW BEST 연출 · 누적 점수** — 기록을 갱신하고 다시 도전하게 만드는 장치.

결과적으로 "안전하게 천천히 클리어" vs "위험하게 빠르게 고득점" 사이의 선택이 곧 플레이 스타일이 된다.

## 설계 결정과 근거

| 결정                                                    | 왜                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **이진 시간 스케일** (멈춤 0.1 / 이동 1.0, 가속 없음)   | 가속 버전을 검토했지만, 속도가 "관리할 부담"이 아니라 "직관적인 리듬"이 되도록 의도적으로 단순화했다.                                                                                                                                                                                                                                                                                                                           |
| **캔버스 게임 + DOM UI 하이브리드**                     | 게임은 Canvas로 렌더하되 HUD·결과·로딩·일시정지는 DOM 오버레이로 분리. 디자인 시스템(토큰·컴포넌트)을 그대로 쓰고 반응형을 확보.                                                                                                                                                                                                                                                                                                |
| **스테이지 요소를 데이터로 고정 (랜덤 생성 제거)**      | 웅덩이·아이템·쉘터를 런타임 랜덤으로 깔았더니 플레이마다 난이도가 달라 유저 간 클리어 공정성이 깨졌다 → 위치를 스테이지 데이터로 고정하고, 비·번개 빈도만 난이도별로 조정.                                                                                                                                                                                                                                                      |
| **스테이지·랭킹을 페이지 → 풀스크린 모달로**            | 페이지 이동 시 BGM이 끊기고 배경 애니메이션이 리셋됐다. 모달로 바꿔 BGM 연속 재생 + 배경 유지 + "닉네임 없으면 진입을 막는 관문"을 메인 한곳에서 처리.                                                                                                                                                                                                                                                                          |
| **진행도를 localStorage → Firestore(`progress/{uid}`)** | 클라이언트가 localStorage를 조작해 스테이지를 임의로 해제하지 못하도록 서버(uid 기준)에서 관리. 인메모리 캐시 + 낙관적 갱신으로 같은 세션에선 즉시 반영.                                                                                                                                                                                                                                                                        |
| **`useSyncExternalStore` 채택**                         | React 19 strict에서 effect 내 동기 `setState`가 cascading 렌더 경고를 유발 → 외부 스토어 구독 패턴으로 회피.                                                                                                                                                                                                                                                                                                                    |
| **에셋 로딩 게이트 + 최소 로딩 시간**                   | 로드 전 플레이스홀더가 비치던 플래시 제거. 캐시로 너무 빨리 끝나도 깜빡이지 않게 최소 3초 로딩 보장.                                                                                                                                                                                                                                                                                                                            |
| **16:9 레터박스 유지 (가변 폭 롤백)**                   | 화면을 꽉 채우려 캔버스 폭을 가변으로 바꿨더니 모바일에서 렌더 부하가 커져 프레임이 떨어졌다 → 성능을 위해 고정 비율로 되돌림. _(시도와 롤백 모두 기록)_                                                                                                                                                                                                                                                                        |
| **세로 기기에서 게임만 CSS 회전**                       | 모바일 플레이 시 "가로로 돌려주세요" 안내 모달로 막는 대신, 게임 캔버스만 CSS로 회전해 세로로 든 채(또는 기기를 돌려) 그대로 플레이하게 했다. 랜딩·랭킹 등 게임 외 화면은 세로 유지. 결과/로딩 등 게임 내 오버레이는 컨테이너 쿼리(cqh)로 비율을 맞춰 가로·세로 어디서든 한 화면에 들어가게 했다. 게임 외 화면도 가로모드에서 깨지지 않게 반응형 처리(메인 버튼은 가로에선 2×2 그리드, 모달은 화면 높이에 맞춰 캡+내부 스크롤). |

## 구조

```
app/                      # 라우트 (App Router)
  page.tsx                # 메인 — 스테이지/랭킹은 모달로 진입
  play/[stageId]/page.tsx # 게임 화면
features/
  game/
    engine/               # 게임 엔진 (상태·물리·렌더) — createEngine
    components/           # GameCanvas · StageGate(잠금 가드) · LoadingOverlay
  stage/                  # 스테이지 데이터·진행도·맵·모달
  ranking/                # 랭킹 저장소·스토어·모달
  user/                   # 익명 인증·닉네임
  settings/ tutorial/     # 설정·게임 방법 모달
components/               # GameButton · Modal · Select · PillTabs · ToggleButton
lib/                      # firebase · sound · orientation · touch · mainView · storage
public/                   # sprites · audio · fonts
```

- **엔진/렌더 분리**: 게임 로직은 순수 모듈(`createEngine`)로 두고, React는 캔버스 마운트·상태 표시만 담당.
- **상태 관리**: 외부 가변 상태(진행도·랭킹·오디오·방향·뷰)는 `useSyncExternalStore` 기반 스토어로 구독.
- **라우팅·모달**: 게임은 별도 라우트, 그 외(스테이지/랭킹/설정/방법)는 메인 위 모달 → BGM 연속성과 "닉네임 없으면 진입을 막는 관문"을 메인에서 일원화.
- **스테이지 진입 가드(`StageGate`)**: 게임 진입 전 서버 진행도를 확인해 잠긴 스테이지는 URL 직접 접근을 막음.
- **데이터 모델 (Firestore)**: `progress/{uid}`(클리어·베스트·누적·시도수), `rankings/{uid}`(닉네임·누적/테마별 점수). 익명 인증(uid)으로 식별.

## 확장성과 운영 관점

단발성 과제가 아니라 **"실제로 운영하며 사용자 리텐션을 키울 서비스"** 를 염두에 두고 설계했다. "한 번 하고 마는 게임"이 아니라 "계속 돌아오게 만드는 구조"가 목표였다.

**진입 마찰을 0에 가깝게** — 익명 인증으로 가입 절차 없이 들어오자마자 플레이할 수 있다. 특히 모바일은 텍스트 입력 자체가 마찰이 커(키보드가 화면을 가리고 입력 UX도 번거롭다) 닉네임 입력 단계를 자동 생성으로 건너뛰게 했다. "가입부터 하라"는 마찰은 첫 세션 이탈의 큰 원인이라, 일단 플레이부터 시키는 걸 우선했다.

**돌아올 이유를 단계마다 심기** — 진입 → 첫 클리어 → 베스트 갱신(NEW BEST 연출) → 랭킹 경쟁 → 다음 테마 기대로 이어지는 흐름을 의도했고, 각 단계가 다음 단계를 끌어당기도록 장치를 배치했다.

- **누적 점수 · 랭킹(누적/테마별)** — 한 번 깬 사람이 "더 잘 깰" 이유. 경쟁과 재도전을 만든다.
- **스테이지 묶음(번들) 구조 + "Coming Soon"** — 현재 **비 오는 거리(4스테이지)** 가 플레이 가능하고, **눈 오는 거리·운석 떨어지는 우주**는 예고 상태. 새 테마를 계속 붙일 수 있는 확장형 구조이자, 다음 콘텐츠를 미리 보여 재방문 동기를 남긴다.

**공정성은 경쟁의 전제** — 랭킹이 의미를 가지려면 모두가 같은 조건에서 겨뤄야 한다. 그래서 웅덩이·아이템·쉘터 위치를 스테이지 데이터로 고정해 플레이어마다 난이도가 달라지지 않게 했다(비·번개 빈도만 난이도별로 조정). 불공정한 랭킹은 경쟁 동기를 꺾고, 그건 곧 리텐션의 누수다.

**다음 단계 (로드맵)**

- Firestore 보안 규칙(uid별 읽기/쓰기 제한) 적용
- 새 테마 맵(눈 오는 거리·우주) 콘텐츠 제작
